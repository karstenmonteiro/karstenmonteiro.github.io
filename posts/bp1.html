What is Docker?

Docker is an open-source platform that gives users the ability to build, test, share, update, manage, and deploy applications quickly and reliably using containers, making development efficient and predictable.


Docker uses containers to package applications, along with all its dependencies and required resources to run it, into lightweight, standardized, executable units of software. Utilized throughout the development lifecycle, Docker simplifies the tedious and repetitive process of configuration, facilitating rapid, effortless, and portable application development. With Docker, you can build and test your applications on your local machine, and then easily and efficiently deploy them to production. Additionally, Docker provides tools for managing and orchestrating containers (which we will discuss more later), making it easier to scale and manage large, complex applications. It also gives developers the ability to deploy multiple, separate containers independently and even in different languages (if desired). Running multiple containers independently helps reduce the risk of conflict between languages, libraries, or frameworks.


Docker works by utilizing the client-server technology, called the docker engine. The docker engine is responsible for running Docker components and services and consists of three primary services, which include:

  • A server that runs the Docker daemon process
  • APIs that are used to communicate and instruct the Docker daemon, and
  • A command-line interface (CLI) for interacting with Docker

The Docker daemon is responsible for creating and managing containers, images, volumes, and networks. Overall, Docker makes it very simple to create and manage containers that can be quickly deployed and scaled into any environment, reliably. Why use Docker?

Docker is widely loved by many developers and major cloud providers and is known to be the best/most popular container management software out there. Major cloud providers such as AWS, GCP, and Azure have all implemented Docker into their systems and also provide support. Because of this, you can seamlessly deploy containers on any environment.

Some benefits that Docker offers include:

  • Improved scalability
  • Reduced operational/infrastructure costs
  • The ability to ship an application anywhere
  • Simplified/rapid software deployment
  • Faster/more efficient testing
  • Portability
  • Efficiency

and many more.


What are Containers?

Containers are portable, lightweight, executable, standardized units of software that isolate an application by packaging it up, along with all of its dependencies and required resources to run it.


Containers are secure. Any applications packaged into a container are automatically safer as Docker is known to provide one of the strongest default isolation capabilities in the industry.


Containers are isolated from other containers to prevent interference, and each container runs its own software, binaries, configurations, etc.


A container can also be described as a runnable instance of an image, as containers are created from container images and are used to build, share, and run your applications. You can create, start, stop, move, or delete a container using the CLI or APIs. Containers run quickly, consistently, and reliably, in any environment, such as:

  • Windows, Mac, and Linux
  • Virutal machines
  • Data centers
  • The cloud
  • Serverless environments
  • Design, Testing, Staging, and Production

and many more. Multiple containers can run on the same machine, each running as an isolated process in the user space and will share the machine’s operating system kernel between each other. Therefore, they do not require an operating system for each application, which increases server efficiency and reduces server and licensing costs.

Container Images

Containers use an isolated custom filesystem when they are running. Container images provide containers with this filesystem and because of this, the image must contain everything that’s required to run your application. This includes all dependencies, configuration settings, binaries, system libraries, utilities, etc. They also contain the environment variables and other metadata. Container images are built in layers, starting with a base image and adding new layers on top of it, if desired, and should be thought of as immutable, as they cannot be changed after they’re created.

Container Volumes

While containers can create, update, and delete, any changes made to the container will be lost when the container is removed. The solution for this is container volumes.


Volumes provide the ability to connect specific filesystem paths in the container with the host machine. Therefore, if a directory in the container is mounted, any changes to it can be seen on the host machine. Volumes can also be used to share files between containers.

Multi-Container Applications

Let’s say you’ve been working on your single container application, and you are now ready to implement a MySQL database. You may be wondering:


"Where do I run MySQL? Do I run it in the same container or do I run it separately?"


The answer is simple. Each container should be focused on doing one thing only. This is because:

  • A container only starts one process, so a process manager would be required to run multiple processes which would then add complexity to the startup/shutdown of containers
  • Having separate containers enables you to version and update versions in isolation
  • It’s most likely that you would have to scale databases differently than other processes

Container Networking

As discussed earlier, Containers run in isolation, so they don’t have the ability to communicate with one another, even if they’re on the same machine. This can be resolved with networking.


If two containers are on the same network, they can communicate with each other. Otherwise, they cannot.


Containers can be added to a network by either:

  • Creating a network and assigning it to a container at startup, or
  • Connecting it to an existing container

Docker Compose

Docker Compose is a tool for managing and running multi-container applications. Docker Compose can:

  • Have multiple isolated environments on a single host
  • Support variables and moving a composition between environments
  • Preserve volume data when containers are created
  • Only recreate containers that have changed

To start with Compose, you need to create a YAML file to configure your application’s services. Then, you create and start the services from the configuration file by executing just a single command. Like Docker, Docker Compose also works in all environments. It also has commands for managing your applications such as:

  • Starting and stopping services
  • Viewing the status/and or streaming the log output of running services
  • Rebuilding services, etc.

Docker CLI Basics

The command line interface (CLI) is used to interact with the Docker engine, which as mentioned above (earlier) consists of three primary services:

  • A server that runs the Docker daemon process
  • APIs that are used to communicate and instruct the Docker daemon
  • A command-line interface (CLI) for interacting with Docker

When a command, i.e., docker container run is executed, the client sends a request through the API, to the Docker daemon, which then handles the request.

Some of the most used commmands include:


Commmand Description
docker image pull <image> Pulls image fromm docker registry.
docker image rm <image> Removes an image.
docker image ls Lists all images.
docker container run <image> Runs a container from an image.
docker container stop <container> Stops a container.
docker container rm <container> Removes a container.
docker container ls -a Lists all containers.
docker container exec <container> Executes a commmand inside the container.
